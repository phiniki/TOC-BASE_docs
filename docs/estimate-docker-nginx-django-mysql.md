# 工数見積メモ：Docker / nginx / Django / MySQL ＋ VPN による開発

**前提**

- **見積対象**：実装・構築・テスト・リリース準備。**要件定義・詳細設計の工数は含めない**（ユーザー指示）。
- **VPN**：本番／ステージングへの**社外からの安全なアクセス**（WireGuard / OpenVPN / クラウド VPN / Zero Trust 系など）の**構築・設定・疎通確認**を**実装工数**として含める。プロバイダ選定の比較検討そのものは最小限に留め、**決まった方式のセットアップ**が中心のイメージ。
- **進め方**：アジャイル想定。スプリント長は **2週間** を仮定して記載（調整可）。
- **現状資産**：`UI/` 配下の静的プロトタイプ（localStorage 前提）。本番相当では **API 化・DB 永続化・認証** が主な追加作業。
- **単位**：**人日（1人がフルタイムで1日）**。複数名時はカレンダー期間が短くなる。

未確定要件が多いため、各ブロックは **控えめ（Min）〜想定（Likely）〜余裕込み（Max）** のレンジで記載。Max は仕様ブレ・既存 HTML との整合・非機能（監視・バックアップ詳細など）のバッファを含むイメージ。

**進め方の前提（重要）**：**常駐型のアジャイル**で進める場合、工数・契約は **「バージョン1（ひとまず使える）」** と **「稼働後（使いながらのバグ・微修正・小改善の積み上げ）」** の **二段階**で見積せよ、が現実的。下記 §2 と、後段の一覧（§3〜）は噛み合わせて読むこと。

## 1. 全体サマリ（ざっくり）

| 区分 | Min（人日） | Likely（人日） | Max（人日） |
|------|------------:|---------------:|------------:|
| **A. 基盤（Docker / nginx / 環境）** | 5 | 8 | 14 |
| **B. Django 基盤・認証・管理画面** | 8 | 14 | 22 |
| **C. MySQL スキーマ・マイグレーション・初期データ** | 6 | 12 | 20 |
| **D. ドメイン API（案件・マスタ・目標・仕入等）** | 25 | 45 | 75 |
| **E. フロント連携（既存 UI の API 接続 or 段階移行）** | 15 | 28 | 45 |
| **F. テスト・品質・ドキュメント（開発者向け）** | 8 | 15 | 25 |
| **G. 本番寄せ（HTTPS・環境変数・バックアップ方針の実装）** | 4 | 8 | 15 |
| **H. VPN（接続基盤の構築・設定・疎通・運用手順のたたき台）** | 4 | 10 | 22 |

## 2. 常駐型アジャイル：バージョン1 と稼働後の二分法

一括見積（§1 の表）は **スコープを全部並べた帯**。実務では、次の **二つに分割**して単価・契約単位・リスクを決めると齟齬が減る。

### フェーズA：バージョン1（MVP／初回リリース可能）

- **意味**：ユーザーが業務として **運用開始できる最小スコープ**（「まず一通り」「主フローを代替」など）までを **ここまでで固定**。以降はフェーズB。
- **含めやすいもの**：§3 の A〜H のうち、**運用開始に必要なセット**（基盤・認証・主要 API・主要画面連携・最低限テスト・本番/VPN の叩き台 等）。機能の細部・非機能の厚みは Likely と Max で **フェーズ B に回すものを明示**するのがポイント。
- **見積の持ち方**：§1 の総額そのままではなく、**V1 スコープに対応した人日**に切り詰める。または V1 で **達成すべきチェックリスト**を先に合意してから、その分解で人日。

### フェーズB：稼働後（使いながらの継続改善）

- **意味**：運用開始後に出る **不具合・表示・文言・ワークフロー微調整**、使ってみて分かった **追加の小機能**。**スプリントを回し続ける**イメージで、フェーズ A のバッファ（Max と Likely の差分）とは別軸になることが多い。
- **見積の持ち方（例）**：
  - **月あたりの拘束日数＋準委任**（例：**月〇人日〜**）。
  - または **時間単位／スプリント定額**（緊急対応 SLA は別）。
- **契約上**：「フェーズ A の受け渡しまでで完結」と書かず、**運用フェーズの予算チャネルを別契約または別項目**として置くことが望ましい。

## 3. ブロック別の内訳（アジャイル向けエピック単位）

### A. 基盤：Docker / nginx / 開発・本番用 compose

- Dockerfile（Django / 静的 or 別フロント）、docker-compose（app / db / nginx）
- 開発：ホットリロード、ボリューム、`.env` テンプレ
- nginx：リバースプロキシ、静的配信、（必要なら）API プレフィックス
- **Min 5 / Likely 8 / Max 14**（本番相当の hardening、マルチステージビルド、CI 用イメージまで含めると Max 側）

### B. Django 基盤

- プロジェクト／アプリ分割方針、設定分離（dev/stg/prod）
- 認証（セッション or JWT）：ログイン・権限ロールの有無で Max が伸びる
- Django Admin の利用範囲（マスタメンテだけ Admin に寄せるか）
- **Min 8 / Likely 14 / Max 22**

### C. MySQL

- 主要エンティティのテーブル定義（案件、担当者、取引先、仕入先、カテゴリ、目標など）
- **仕入明細**：案件への参照は **NULL 可**（案件に紐づかない仕入行を許容）。一覧では「案件未紐づけのみ」などで絞り込み可能にする想定。
- マイグレーション運用（Django migrations）
- 既存 JSON／localStorage からの移行スクリプトの要否（**要ると D・E と合わせて +5〜15人日** 程度のバッファが現実的）
- **Min 6 / Likely 12 / Max 20**

### D. ドメイン API（REST 想定）

現行 UI の機能をバックエンド化するイメージでの粗い積み上げ（**画面1本ずつではなくドメイン単位**）。

- 案件 CRUD・一覧・フィルタ・請求済み更新
- 完了シート用の集計 or クエリ
- 各種マスタ CRUD
- 目標設定・（あれば）ダッシュボード用集計 API
- 仕入・**任意の**案件紐付け（NULL 可）・原価集計（案件別は紐づく行のみ、またはルールに応じて未紐づけを別集計）
- 仕入 CRUD（登録画面：案件単位・請求書単位・個別行）。一覧専用画面は UI では持たない想定
- 権限・バリデーション・楽観ロックの要否で **Max が大きく伸びる**

**Min 25 / Likely 45 / Max 75**

### E. フロント連携

- 既存 `UI/` を **静的のまま nginx 配信 + Fetch で API** に切り替え（段階的にモジュール化）
- または一部テンプレートを Django に寄せる場合は工数再配分
- CORS、CSRF、認証トークン／Cookie の扱い
- **Min 15 / Likely 28 / Max 45**

### F. テスト・品質

- API の最低限の自動テスト（pytest）、CI でのテスト実行
- 主要フローの手動試験観点の整理（チェックリスト程度は実装に含める想定）
- **Min 8 / Likely 15 / Max 25**

### G. 本番寄せ

- TLS 終端（nginx + Let’s Encrypt 等）、ログ、簡易バックアップ（DB ダンプの cron 等）
- **Min 4 / Likely 8 / Max 15**

### H. VPN

**想定**：開発・検証環境または本番への**閉域アクセス**（インターネットに素直に晒さない、または管理ポートを VPN 内に閉じる等）。

- VPN 方式の**実装**：サーバー（またはゲートウェイ）側のデーモン・鍵・ファイアウォール／セキュリティグループ
- **ルーティング**：分割トンネル／フルトンネル、許可する宛先（SSH・HTTPS・DB を VPN 内のみにするか等）
- **クライアント**：設定ファイルまたはインストーラ手順のたたき台、**数名分の試験接続**まで
- **運用メモ**：証明書・鍵の更新、利用者追加の手順（詳細設計は含めず「手順の素案」レベル）

**Min 4 / Likely 10 / Max 22**

- **Max が伸びる例**：拠点間冗長、既存オフィス機器との IPsec 連携、MFA・デバイス証明まで含める、監査ログ要件、クライアント数十台規模の個別発行
- **Managed サービス**（Tailscale / Cloudflare Tunnel 等）に寄せるとサーバー自作より **短くなることもある**（その場合は H を **Likely 5〜8人日** 程度に再配分する想定）

## 4. アジャイルでの進め方（例）

- **§2 と揃える**：**フェーズA（バージョン1）のゴール**までが「運用開始」スコープ。そこまでに含めるストーリーを先に並べ、その後は **フェーズB** でバックログを処理すると、§1 の一括総日数（Likely **140** など）が **単期契約だけの見込みにならず**済む。
- **スプリント 0（0.5〜1スプリント）**：A の大半 + B・C の足場（「動くパイプ」）。**G・H は並行または直後のスプリントで着手**（本番 URL が決まるタイミングと合わせる）
- **以降**：D のエピックを優先度順に分割（例：案件 → マスタ → 目標 → 仕入 → ダッシュボード）
- 各スプリントで **E を少しずつ前進**（1画面 or 1機能の API 化）し、デモ可能なインクリメントにする
- **Likely 合計 140人日 ÷ 10営業日/スプリント ≈ 14スプリント分の「純作業」** → 並行・割込みを考慮し **カレンダーは余裕を見る**

## 5. 見積を上下させる主な要因

| 要因 | 影響 |
|------|------|
| 認証・権限（ロール・行レベル）を厳密にやる | D・B が **+30%〜** になりやすい |
| 完了シート・ダッシュボードの集計を SQL 重めで一発 | C・D で **+5〜20人日** |
| 既存 localStorage データの本番移行ツールが必要 | **+5〜15人日** |
| フロントを SPA 化（Vue/React）に寄せる | E が **Max を超えて別見積** になりやすい |
| 監査ログ・複数テナント | 別エピックとして **+10人日〜** |
| VPN を**拠点冗長・大規模クライアント・厳格コンプラ**まで含める | **H が +10人日〜**（Max 帯） |
| **Managed VPN / Tunnel** に寄せて自作を最小化 | **H が −3〜5人日** 程度になることあり |
